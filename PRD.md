# 产品需求文档 — AI Pull Request 描述生成器

## 1. 产品概述

### 1.1 产品定位
一款 GitHub App，当开发者创建/更新 Pull Request 时，自动分析代码变更（diff），生成结构化的 PR 标题、描述、变更摘要、测试建议和影响范围说明。

### 1.2 核心假设
- 开发者普遍讨厌写 PR 描述，但好的 PR 描述对团队协作至关重要
- GitHub App 形态「安装即用」，零配置，降低使用门槛
- LLM 分析 diff 生成描述的准确度足够高（>80% 可直接使用或微调）

### 1.3 目标用户画像

| 用户类型 | 描述 | 核心需求 | 付费意愿 |
|---------|------|---------|:-------:|
| **个人开发者** | 独立开发者、开源贡献者 | 快速生成 PR 描述，少写文档 | 低-中 |
| **小微团队**（2-10 人） | Startup、自由职业团队 | 统一 PR 规范，提升 Code Review 效率 | 中 |
| **中型团队**（10-50 人） | 成长型 SaaS 公司 | 自动测试建议 + 影响范围分析 | 高 |
| **企业团队**（50+ 人） | 成熟科技公司 | 合规、审计、自定义流程 | 很高 |

## 3. 功能规格

### 3.1 GitHub App 核心流程

```
PR Created/Updated → Webhook Received → Fetch PR Diff
    → Call LLM API → Generate Description → Post Comment
```

### 3.2 AI 生成内容结构

```markdown
## 📋 PR 描述 (由 AI 自动生成)

### 标题
[简明的标题，基于变更内容]

### 变更摘要
- **做了什么：** [简要描述变更内容]
- **为什么做：** [推理变更原因]
- **影响范围：** [涉及的文件/模块]

### 变更详情
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| src/foo.ts | 修改 | 添加了 X 功能 |
| src/bar.ts | 新增 | 实现了 Y 接口 |

### 🧪 测试建议
- [ ] 单元测试：验证 X 功能在边界条件下的行为
- [ ] 集成测试：确认 Y 接口与下游服务的兼容性

### ⚠️ 影响范围
- **破坏性变更：** 否/是（说明原因）
- **需要更新的文档：** README.md
- **相关 Issue：** #123

---

> 🤖 此描述由 AI 自动生成，请审核后使用。
```

### 3.3 触发条件

| 事件 | 动作 |
|------|------|
| `pull_request.opened` | 首次生成 PR 描述 |
| `pull_request.synchronize` | 有新的 commit push，重新生成 |
| `pull_request.edited` | 如果 body 为空，重新生成；否则跳过 |

### 3.4 限流与防滥用

- 每个 PR 最多生成 5 次（防止循环触发）
- 每次生成间隔至少 30 秒
- 每天每个仓库最多生成 100 次

## 4. 非功能需求

| 需求 | 指标 |
|------|:----:|
| **延迟** | PR 创建到 AI 评论生成 < 10 秒 |
| **可用性** | 99.5%（核心 Webhook 处理） |
| **LLM 生成质量** | 80%+ 的生成描述不需大幅修改即可使用 |
| **安全** | 不存储用户代码，diff 处理后不保留 |
| **成本控制** | 每次 PR 生成 LLM API 成本 < $0.01 |

## 5. 边界与排除

| 边界 | 说明 |
|------|------|
| **第一个 MVP 不做** | 企业 SSO、私有化部署、自定义模板、Web Dashboard |
| **不支持的场景** | GitLab、Bitbucket（后续版本考虑） |
| **不分析的场景** | 超过 500 个文件的巨型 PR（跳过，提示人工处理） |
