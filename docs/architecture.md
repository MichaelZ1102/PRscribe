## 2. 核心组件说明

### 2.1 Webhook Handler
- **职责：** 接收 GitHub 发出的 Webhook 事件，验证签名，去重，限流，路由
- **关键依赖：** `@octokit/webhooks`（仅用 verify 方法，不依赖 Express 中间件）
- **处理流程：**
  1. 验证 `x-hub-signature-256` 签名 → 非法请求返回 400
  2. 检查 `x-github-delivery` 是否已处理（Redis Set）→ 已处理返回 200（幂等）
  3. 检查限流 → 超限返回 429
  4. 路由到对应处理器
  5. **立即返回 202 Accepted**，后续异步处理
  6. 将任务推入 Redis Queue

### 2.2 幂等性去重
- **原因：** GitHub Webhook 在未收到 200 时会重试（1min → 5min → 30min）
- **方案：** 用 Redis Set 存储已处理的 `x-github-delivery`
- **TTL：** 24 小时
- **注意：** 即使 LLM 调用成功但写评论失败（返回非 200），幂等性会阻止重入。此时需要人工介入或重试队列。

### 2.3 Diff Fetcher
- **职责：** 通过 GitHub API 获取 PR 的 diff 内容
- **API 端点：** `GET /repos/{owner}/{repo}/pulls/{pull_number}`
- **响应头：** `Accept: application/vnd.github.diff`
- **限制处理：** diff > 500 文件或 > 5MB 时跳过，返回提示

### 2.4 Prompt Builder
- **职责：** 将 diff 内容组装成 LLM 提示词模板
- **核心逻辑：**
  1. 提取变更文件列表（路径、变更类型）
  2. 提取变更代码片段（按文件分组，有上下文）
  3. 组装到预设的 prompt 模板中
  4. 计算 token 数，超过限制时自动截断（优先保留变更摘要）

### 2.5 LLM Client
- **职责：** 调用 LLM API 生成 PR 描述
- **MVP 策略：** 仅用 GPT-4o-mini（成本低、速度快）
- **后续迭代：** 升级到 GPT-4o（质量不够时）、Claude Sonnet 4（大 diff）
- **重试策略：** 2 次重试，指数退避（3s → 9s）
- **超时限制：** 15 秒

### 2.6 GitHub API Client
- **职责：** 将生成的描述以评论形式发布到 PR 中
- **API 端点：** `POST /repos/{owner}/{repo}/issues/{issue_number}/comments`
- **同时更新 PR body（可选）：** `PATCH /repos/{owner}/{repo}/pulls/{pull_number}`

## 3. 数据流

### 3.1 正常流程

```
1. 开发者在 GitHub 创建 PR
2. GitHub 发送 pull_request.opened webhook 到我们的服务器
3. Webhook Handler 验证签名，确认事件合法
4. 检查 x-github-delivery 是否已处理（幂等性）
5. 检查限流
6. 路由事件到处理器
7. 立即返回 202 Accepted（不阻塞）
8. 将处理任务推入 Redis Queue
9. 队列消费者出队任务
10. Diff Fetcher 通过 GitHub API 获取 PR diff
11. 检查 diff 大小，超过限制则跳过
12. Prompt Builder 将 diff 组装为 LLM prompt
13. LLM Client 调用 LLM API 生成描述
14. 解析 LLM 返回的结构化内容
15. GitHub API Client 将描述以评论形式发布到 PR
16. 记录用量日志到 Redis
17. 记录 delivery_id 已处理（防止重放）
```

### 3.2 错误处理

```
Webhook 签名验证失败 → 400（GitHub 会重试 3 次，但签名失败不重试）
幂等性命中 → 200（已处理过，忽略）
限流命中 → 429（GitHub 会重试）
LLM API 超时 → 重试 2 次，均失败 → PR 评论"生成失败，请重试"
Diff 太大无法处理 → PR 评论"变更过大，无法自动生成，请手动编写"
生成内容为空 → PR 评论"无法分析变更内容，建议手动编写"
队列消费失败 → 记录到 Sentry，不重试（避免循环消费）
```

## 4. 部署架构（MVP）

### 推荐方案：Railway / Fly.io（自托管 Node.js）

```
┌───────────────────────────────────────────────────────────┐
│                  Railway / Fly.io                          │
│  ┌────────────────────────────────────────────────────┐   │
│  │  Node.js (Hono.js) Server                          │   │
│  │  - POST /api/v1/webhook (接收 GitHub Webhook)     │   │
│  │  - GET /api/v1/health (健康检查)                   │   │
│  │  - Bull Queue Worker（后台消费任务）               │   │
│  └────────────────────────────────────────────────────┘   │
│  ┌────────────────────────────────────────────────────┐   │
│  │  Upstash Redis                                     │   │
│  │  - 任务队列 (Bull)                                 │   │
│  │  - 幂等性去重                                      │   │
│  │  - 限流计数                                        │   │
│  │  - 用量跟踪                                        │   │
│  └────────────────────────────────────────────────────┘   │
│  ┌────────────────────────────────────────────────────┐   │
│  │  Sentry                                            │   │
│  │  - 错误追踪                                        │   │
│  │  - 性能监控                                        │   │
│  └────────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────────┘
```

**弃用 Vercel Serverless Hobby Plan 的原因：**
| 限制 | Vercel Hobby | 要求 |
|:----|:------------|:-----|
| 函数超时 | 10s | LLM 调用需要 15-20s |
| 并发请求 | ？ | 队列需要在进程内运行 |
| In-Memory | 无状态 | 需要持久队列 |

### 备选方案（如需 Vercel）
升级到 Vercel Pro（$20/月）获得 60s 超时 + 使用 Vercel KV（Redis）做队列。

## 5. 安全设计

| 安全项目 | 方案 |
|---------|------|
| **Webhook 签名验证** | 使用 GitHub App 的 Webhook Secret 验证 `x-hub-signature-256` |
| **GitHub API 认证** | 使用 GitHub App 的 Private Key 生成 JWT，换取 Installation Token |
| **用户代码安全** | LLM 调用后即刻丢弃 diff 内容，不落盘 |
| **API Key 管理** | 所有 API Key 存储在环境变量中 |
| **HTTPS 强制** | 所有通信均通过 HTTPS |
| **幂等性** | Redis 记录 delivery_id，24h 有效期，防止 webhook 重放 |
